### Q1. JWT 인증 방식을 선택한 이유는? 세션/쿠키 대비 장단점은?

**A:** 과제 요구사항에서 JWT 인증 방식을 더 고민해볼 키워드로 제시해서 도전해봤습니다. 세션 방식은 서버에 세션 저장소가 필요하지만, JWT는 토큰 자체에 사용자 정보가 담겨 있어 서버가 상태를 저장할 필요가 없습니다(stateless). 다만 토큰 탈취 위험이 있어서 HTTP-only 쿠키에 저장하고 만료 시간을 60분으로 설정했습니다. 단점으로는 토큰 발급 후 강제 만료가 어렵다는 점이 있습니다.

### Q2. Bootstrap을 사용했는데, 다른 CSS 라이브러리(Tailwind, Bulma 등)는 고려하지 않았나?

**A:** 과제 요구사항에서 대체 라이브러리도 고민해보라고 했지만, 3박 4일이라는 짧은 기간 안에 완성도를 높이는 것이 우선이었습니다. Bootstrap은 팀원 모두 입학시험에서 사용해본 경험이 있어 학습 비용 없이 바로 개발에 집중할 수 있었습니다. 시간이 더 있었다면 Tailwind를 써봤을 것 같습니다.

---

### Q3. Jinja2 서버사이드 렌더링을 어떻게 활용했나?

**A:** 모든 페이지를 Jinja2 템플릿으로 렌더링합니다. base.html이라는 공통 레이아웃을 만들고 각 페이지가 이를 상속받는 구조입니다. Flask의 context_processor를 사용해서 모든 템플릿에 로그인 사용자 정보를 자동 주입하고, 로그인 여부에 따라 네비게이션 바 내용을 분기합니다.

---

### Q4. MongoDB를 선택한 이유는? 스키마 설계는 어떻게 했나?

**A:** 과제 요구사항에서 MongoDB를 지정했습니다. 스키마 설계에서는 프로젝트 문서 안에 테스트케이스 배열을 임베딩하고, 각 테스트케이스 안에 피드백 배열을 임베딩하는 구조를 택했습니다. 프로젝트를 조회할 때 관련 데이터를 한 번의 쿼리로 가져올 수 있어서 성능상 이점이 있습니다. 다만 중첩이 깊어서 업데이트가 복잡해지는 트레이드오프가 있었습니다.

### Q5. AWS 배포는 어떻게 진행했나?

**A:** 팀원 한 명의 EC2 인스턴스에 Docker Compose로 Flask 앱과 MongoDB를 함께 올렸습니다. GitHub Actions로 CI/CD 파이프라인을 구성해서, main 브랜치에 push하면 SSH로 EC2에 접속해 자동으로 git pull, 컨테이너 재빌드, 재시작이 됩니다. 도메인은 별도로 붙이지 않았습니다.

### Q7. 기능 범위를 어떻게 설정했나? 왜 이 범위인가?

**A:** 과제 가이드에서 "기능과 범위를 좁게 유지하라"고 했기 때문에, 핵심 기능인 프로젝트 CRUD, 테스트 케이스 기반 피드백, 회원 인증 세 가지에 집중했습니다. 처음에는 댓글이나 알림 기능도 논의했지만, 작은 범위를 완성도 높게 만드는 것을 우선했습니다.

### Q8. 와이어프레임은 어떤 도구로 작성했고, 어떤 과정을 거쳤나?

**A:** (팀 상황에 맞게 작성 - 예: Figma/PPT/노트 등) 첫날 팀원 모두 모여서 화면 구성을 잡고, 각 페이지에 필요한 API를 함께 정리했습니다. 와이어프레임에 시간을 오래 쓰지 않고 빠르게 합의 후 구현에 들어갔습니다.

### Q9. 로그인 기능에서 보안적으로 신경 쓴 부분은?

**A:** 비밀번호는 bcrypt로 해시하여 저장하고 평문은 어디에도 보관하지 않습니다. JWT 토큰은 HTTP-only 쿠키에 저장해서 JavaScript에서 접근할 수 없게 했고, 만료 시간을 60분으로 제한했습니다. 회원가입 시 비밀번호 최소 8자 이상, 비밀번호 확인 일치 검증도 적용했습니다.

### Q10. 입학시험에서 배운 기술 외에 추가로 사용한 기술이 있는가?

**A:** Docker와 GitHub Actions를 활용한 CI/CD 파이프라인은 입학시험 범위를 넘어서는 부분입니다. 다만 과제에서 금지한 React, Vue, MySQL 같은 프레임워크나 DB는 사용하지 않았고, 요구사항에서 제시한 기술 스택 안에서 배포 자동화를 추가한 정도입니다.
